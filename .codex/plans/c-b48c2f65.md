# Codex plan for mehrangta/funny-whiteboard-paint#3

Job: `c-b48c2f65` · Revision: 1

Add separate Windows x64 CI and release workflows for the Tauri 2 application. CI will reproducibly install locked Bun and Cargo dependencies, compile the frontend and Rust backend, build MSI and NSIS packages, and retain them as workflow artifacts. Stable version tags will validate all repository versions before creating a draft GitHub Release containing the Windows installers. The plan also repairs the stale Bun lockfile metadata, hardens action permissions and version pinning, documents the release procedure, and adds a manual Windows smoke-test gate before publication.

## Validation

- Run `bun install --frozen-lockfile`; it must complete without modifying `bun.lock`.
- Run `bun run build`; TypeScript strict checks and the Vite production build must succeed.
- On Windows x64, run `cargo check --manifest-path src-tauri/Cargo.toml --locked --target x86_64-pc-windows-msvc`.
- On Windows x64, run `bun run tauri build -- --target x86_64-pc-windows-msvc --bundles nsis,msi`.
- Assert with PowerShell that one non-empty NSIS `*-setup.exe` and one non-empty MSI file exist beneath the target-specific `bundle` directory.
- Open a pull request and confirm it receives no write-capable token, produces the named Windows artifact, and cannot invoke the release workflow.
- Push a disposable matching version tag in a test repository or controlled rollout and verify version mismatches fail before a release is created.
- Install, launch, exercise drawing and save/load, and uninstall both MSI and NSIS outputs on a clean Windows x64 machine; no automated runtime or UI tests currently exist in the repository.

## Risks

- `bun.lock` currently records workspace name `paint_test` while `package.json` uses `funny_whiteboard`; frozen installation may fail until the lockfile is regenerated with the pinned Bun version.
- The application version is duplicated across three files, and the existing `1.0` tag does not match version `0.1.0`; the release gate must reject mismatches before uploading assets.
- Unsigned MSI and NSIS packages can trigger Windows SmartScreen or antivirus reputation warnings; signing later will require a certificate provider and protected credentials or OIDC configuration.
- The repository has no automated Rust, frontend, integration, or UI tests, so a successful installer build does not prove drawing, transparency, filesystem, or WebView behavior.
- Windows installer builds are comparatively slow and depend on runner-provided Visual Studio, WiX/NSIS support, and WebView tooling; bounded caching and timeouts reduce but do not eliminate runner-image failures.
- Mutable runner images and action tags can introduce unreviewed changes; full-SHA pins plus Dependabot reduce supply-chain and compatibility risk.
- Hard-coded artifact filenames would break whenever the application version changes; validation and upload paths must use constrained MSI and NSIS globs.
- A failure during release asset upload can leave a partial GitHub Release; keeping releases as drafts until post-build validation prevents incomplete public distribution.
