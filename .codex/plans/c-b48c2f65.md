# Codex plan for mehrangta/funny-whiteboard-paint#3

Job: `c-b48c2f65` · Revision: 2

Set up separate Windows x64 CI and release workflows that build the Tauri application in release mode without bundling installers. Pull requests and `main` pushes will compile and retain the unsigned executable as a short-lived workflow artifact. Stable version tags will validate repository versions, build exactly one plain `.exe`, and attach it directly to a GitHub Release—using a draft release pending the remaining publication-policy decision. No MSI, NSIS, ZIP, companion mascot file, signing credentials, or installer behavior will be added.

## Steps

1. **Normalize locked build inputs** — Add `.bun-version` pinned to Bun `1.3.14`, matching the re-inspected environment, and use that version throughout local and GitHub builds. Regenerate `bun.lock` without intentionally upgrading dependencies so its stale workspace name changes from `paint_test` to the actual `package.json` name `funny_whiteboard`; verify a subsequent `bun install --frozen-lockfile` produces no changes. Keep `src-tauri/Cargo.lock` committed and pass Cargo's `--locked` option everywhere. Use the stable Rust toolchain with the explicit `x86_64-pc-windows-msvc` target. Do not add invented test or lint scripts; the repository currently defines only frontend build and Tauri commands.
   - Likely files: `.bun-version`, `bun.lock`
2. **Create Windows x64 CI workflow** — Create `.github/workflows/ci.yml` triggered by pull requests, pushes to `main`, and `workflow_dispatch`. Define one stable job/check name such as `Windows x64 executable`, run it on `windows-latest`, set a finite timeout, grant only `contents: read`, and cancel superseded runs for the same branch or pull request. Check out without persisted credentials, install Bun from `.bun-version`, install stable Rust with `x86_64-pc-windows-msvc`, optionally restore a bounded Cargo cache, and run `bun install --frozen-lockfile`, `bun run build`, and `cargo check --manifest-path src-tauri/Cargo.toml --locked --target x86_64-pc-windows-msvc`.
   - Likely files: `.github/workflows/ci.yml`
3. **Build only the plain executable** — In CI, run `bun run tauri build --target x86_64-pc-windows-msvc --no-bundle`. The Tauri CLI's `--no-bundle` option skips configured MSI and NSIS generation while preserving the existing `bundle.targets: "all"` behavior for developers who deliberately run the default local build; therefore `src-tauri/tauri.conf.json` does not need a compatibility-changing edit. The expected output is `src-tauri/target/x86_64-pc-windows-msvc/release/funny_whiteboard.exe`.
   - Likely files: `.github/workflows/ci.yml`, `src-tauri/tauri.conf.json`
4. **Validate and retain CI output** — Use PowerShell after the build to require the expected `.exe` to exist, have a non-zero size, and report `NotSigned` from `Get-AuthenticodeSignature`, making the unsigned decision explicit. Compute its SHA-256 hash and write the filename, byte size, and hash to the GitHub job summary. Copy it into a clean staging directory as `funny-whiteboard-windows-x64.exe` and upload only that file as workflow artifact `funny-whiteboard-windows-x64-${{ github.sha }}` with approximately 14 days of retention. Do not upload `target` directories, `.pdb` files, MSI/NSIS directories, a ZIP assembled by the workflow, or `held-whiteboard.png`; the frontend already embeds the fallback mascot and only overrides it when an optional adjacent PNG exists.
   - Likely files: `.github/workflows/ci.yml`, `index.html`, `src/main.ts`, `src-tauri/src/lib.rs`
5. **Create version-gated release workflow** — Create `.github/workflows/release.yml` triggered only by pushed tags matching `v*`. Use a per-tag concurrency group without cancellation, no pull-request trigger, and job-level `contents: write`; all setup and build actions otherwise receive the minimum permissions. Before building, require a stable tag matching `^v[0-9]+\.[0-9]+\.[0-9]+$`. Read the frontend and Tauri configuration versions through PowerShell JSON parsing and obtain the crate version through `cargo metadata --no-deps --format-version 1 --manifest-path src-tauri/Cargo.toml`. Fail before release creation unless `package.json`, `src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml`, and the tag without `v` all match exactly. This prevents the existing legacy tag `1.0` from being treated as version `0.1.0`.
   - Likely files: `.github/workflows/release.yml`, `package.json`, `src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml`
6. **Publish one direct EXE asset** — Repeat the locked Windows x64 setup and `--no-bundle` build in the release workflow, validate the unsigned executable, and rename a staged copy to `funny-whiteboard-${version}-windows-x64.exe`. Use the runner's GitHub CLI with `GH_TOKEN: ${{ github.token }}` and `--verify-tag` to create a generated-notes GitHub Release and attach exactly that `.exe`; under the recommended policy, include `--draft`. On a rerun, update only an existing draft for the same tag and upload the same asset name with `gh release upload --clobber`; abort rather than mutate an already published release. GitHub CLI supports verified-tag draft creation, direct asset attachment, and clobbering an existing draft asset. ([cli.github.com](https://cli.github.com/manual/gh_release_upload?utm_source=openai))
   - Likely files: `.github/workflows/release.yml`
7. **Harden workflow dependencies** — Pin `actions/checkout`, `oven-sh/setup-bun`, the selected Rust toolchain action, any Rust cache action, and `actions/upload-artifact` to full commit SHAs, with comments identifying their reviewed release lines; current GitHub-hosted examples use `actions/checkout@v6`, and the current artifact action line is v7. Add `.github/dependabot.yml` with a weekly `github-actions` update entry so SHA updates arrive as reviewable pull requests. Keep CI read-only, give release write access only to repository contents, avoid `pull_request_target`, and use no signing, certificate, PAT, or application secrets. GitHub identifies a full-length commit SHA as the immutable way to pin an action.
   - Likely files: `.github/workflows/ci.yml`, `.github/workflows/release.yml`, `.github/dependabot.yml`
8. **Document executable release behavior** — Update `README.md` to add an automated-release section stating that Windows x64 releases contain one unsigned executable and require no installation: download it to a writable directory, acknowledge any SmartScreen warning, run it, and delete it to remove the application. Retain the existing Windows WebView2 runtime prerequisite because the plain executable does not install missing system components. Distinguish the automated `--no-bundle` command from the current default local build, document the exact asset naming convention, and clarify that `held-whiteboard.png` remains an optional adjacent override rather than a required release file. Document version synchronization and the `vX.Y.Z` tag process, including that legacy tag `1.0` must not be moved or reused.
   - Likely files: `README.md`

## Risks

- `bun.lock` still identifies the workspace as `paint_test` while `package.json` uses `funny_whiteboard`; frozen installation may fail until the lockfile is regenerated with the pinned Bun version.
- A plain Tauri executable does not bootstrap WebView2 or other missing Windows prerequisites, so it may not run on a clean system lacking the runtime that the repository already lists as required.
- The executable is intentionally unsigned and can trigger SmartScreen, antivirus reputation, or corporate application-control warnings; the README and release notes must state this clearly.
- Tauri describes plain-binary uploading as a portable-mode workaround rather than a fully supported installer path, so clean-machine smoke testing is essential before publication.
- The repository has no automated Rust, frontend, integration, or UI tests; successful compilation does not prove transparent-window rendering, filesystem access, drawing behavior, or WebView compatibility.
- The application version is duplicated across three files, while legacy tag `1.0` conflicts with current version `0.1.0`; release gating must remain in place before any write operation.
- GitHub runner images and mutable action references can change behavior; full-SHA action pins and Dependabot reduce, but do not eliminate, runner and supply-chain drift.
- Using `--clobber` on a retry deletes the old same-named draft asset before re-uploading it, so an interrupted retry can temporarily leave the draft without an asset; published releases must never be clobbered automatically. ([cli.github.com](https://cli.github.com/manual/gh_release_upload?utm_source=openai))
