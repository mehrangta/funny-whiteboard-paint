Job: `c-b48c2f65` · Revision: 3

Implement two GitHub Actions workflows for the existing Tauri 2 application: a read-only Windows x64 CI workflow for pull requests and `main`, and a tag-triggered release workflow that builds exactly one unsigned plain `.exe`, creates a draft GitHub Release, and leaves publication to a maintainer after clean-machine testing. ## Risks

- `bun.lock` currently names the workspace `paint_test`; frozen installation may fail until it is regenerated with the pinned Bun version, and regeneration must be reviewed to ensure it does not introduce unrelated dependency upgrades.
- The plain executable does not install WebView2 or other missing system prerequisites, so it may fail on Windows systems without the runtime already present; README and release notes must retain this prerequisite.
- The executable is intentionally unsigned, so SmartScreen, antivirus reputation systems, or organizational application controls may warn or block it; this is expected for the initial release and must be documented.
- A no-bundle executable is not an installer-backed distribution path; clean-machine testing is required because automation cannot prove system-runtime compatibility or application startup behavior.
- The repository has no automated Rust unit, frontend, integration, or UI tests, so successful compilation and PE validation do not prove transparent rendering, drawing correctness, filesystem access, or WebView behavior.
- Versions are duplicated across `package.json`, `src-tauri/Cargo.toml`, and `src-tauri/tauri.conf.json`, while legacy tag `1.0` conflicts with current version `0.1.0`; the release gate is mandatory and must execute before any write-capable release step.
- Using `gh release upload --clobber` on a draft retry removes the prior same-named asset before replacement; an interrupted retry can temporarily leave the draft without an executable, but the release remains non-public and the post-upload asset assertion will fail visibly.
- GitHub-hosted runner images and action dependencies can change; full-SHA pins, Dependabot updates, lockfiles, explicit architecture validation, and manual smoke testing reduce but cannot eliminate that drift.
