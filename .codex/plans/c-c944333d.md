# Codex plan for mehrangta/funny-whiteboard-paint#1

Job: `c-c944333d` · Revision: 1

Implement a frontend-only Text tool that renders committed text directly into the existing raster canvas, so text automatically works with PNG save/load and Rust-backed snapshot history without backend or file-format changes. The draft assumes the recommended interaction: users click the board to open an inline multiline editor, format text with shared color plus text-specific size, bold, and italic controls, commit with Ctrl/Cmd+Enter, and cancel with Escape.

## Steps

1. **Add Text Tool Markup** — In `index.html`, add `text` to the equipment tool buttons with the same active-tool semantics as brush, line, and rectangle. Introduce a board-layer wrapper containing the canvas and an initially hidden inline text editor so the editor can be positioned and clipped within the drawable whiteboard. Add text-format controls for bold and italic using toggle buttons with `aria-pressed`; retain the existing color picker as the shared drawing/text color and reuse the size control as a context-sensitive stroke-size or font-size control.
   - Likely files: `index.html`
2. **Model Text Formatting State** — In `src/main.ts`, extend `Tool` with `text` and add explicit text state for font size, bold, and italic while preserving the existing stroke size independently. When tools change, reconfigure the shared size slider and readout without losing either stored value; retain the existing 1–30 px range for strokes and use a practical text range such as 8–96 px with a 32 px default. Update bold and italic buttons, `aria-pressed`, active styling, and editor preview whenever formatting changes. Color remains shared by all tools.
   - Likely files: `src/main.ts`, `index.html`
3. **Implement Inline Editing Lifecycle** — Add a `TextSession` model containing the board anchor, text value, and current style. When the active tool is `text`, canvas pointer-down must bypass drawing capture and open or reposition the inline editor at the clicked board coordinate. Keep the editor inside the board, size its available width to the remaining drawable area, preserve explicit newline characters, and avoid automatic wrapping unless wrapping is separately requested. Ctrl/Cmd+Enter commits, Escape cancels, and Enter alone inserts a newline. Ignore commit shortcuts while an IME composition is active. Clicking another board location or switching tools commits non-empty text before starting the next action; empty or whitespace-only sessions close without changing history.
   - Likely files: `src/main.ts`, `index.html`
4. **Render Text Into Canvas** — Centralize text completion into idempotent `commitTextSession` and `cancelTextSession` helpers. On commit, use `ctx.save()`, the selected fill color, `textBaseline = "top"`, and a canvas font string assembled from italic, bold, font size, and the editor's computed font family. Split text on explicit newlines and call `fillText` for each line using a consistent explicit line height, then restore the context. Canvas coordinates remain CSS-pixel coordinates because `applyCanvasDefaults` already applies the device-pixel-ratio transform. Push exactly one snapshot after a successful non-empty commit; never push a snapshot for cancellation or empty input.
   - Likely files: `src/main.ts`
5. **Integrate Existing Canvas Actions** — Route brush, line, and rectangle pointer events through their existing paths unchanged, while text uses the new editor path. Before save, commit pending non-empty text so exported PNGs include it. Clear and load should cancel uncommitted text before replacing the board. If undo is pressed during editing, first cancel the uncommitted session without consuming canvas history; subsequent undo presses use the existing Rust command. Apply the same cancellation rule before redo. Keep formatting controls usable while the editor is open rather than treating toolbar focus changes as an implicit commit. Reposition and clamp an active editor during canvas resize using its normalized board anchor, without creating a snapshot.
   - Likely files: `src/main.ts`
6. **Style Responsive Controls** — In `src/styles.css`, move the existing board position and dimensions to the new board-layer wrapper, make the canvas fill that wrapper, and clip the inline editor to the drawing area. Give the editor a visible editing boundary and caret while matching the canvas font family, size, weight, style, color, and line height. Use a text cursor for the Text tool and retain the crosshair for drawing tools. Update the equipment grid for the added tool, add compact bold/italic control styling and visible keyboard focus states, and verify the existing breakpoints above 940 px, from 561–940 px, and at 560 px or narrower do not overflow or obscure controls.
   - Likely files: `src/styles.css`, `index.html`, `src/main.ts`
7. **Document And Validate Rollout** — Update the feature list and usage notes to explain text placement, sizing, coloring, bold/italic toggles, commit, cancellation, and the fact that committed text becomes part of the bitmap. No Rust command, capability, dependency, migration, or saved-file format change is required; existing PNG files continue loading normally, and newly saved PNGs remain readable by previous app versions. Validate frontend compilation first, then exercise full history and persistence in Tauri development mode before release packaging.
   - Likely files: `README.md`

## Validation

- Run `bun install` because dependencies are absent in the inspected worktree, then run the repository-defined `bun run build` to validate strict TypeScript compilation and the Vite production build.
- Run the repository-defined `bun run dev` for rapid layout and Canvas rendering checks; expect Rust-backed undo, redo, save, load, and runtime mascot commands to warn and return null in browser-only mode.
- Run the repository-defined `bun run tauri dev` for end-to-end validation of text commits, undo/redo, clear, save, load, and window resizing.
- Verify plain, bold, italic, and bold-italic text at minimum, default, and maximum font sizes and with multiple colors; confirm the inline editor and final canvas output use matching placement and styling.
- Verify multiline text preserves explicit line breaks; Ctrl/Cmd+Enter creates one history entry, Escape and whitespace-only input create none, and Enter during IME composition does not accidentally commit.
- Verify undo removes one committed text block, redo restores it, undo during an uncommitted session only cancels that session, save includes pending non-empty text after committing it, and clear/load discard uncommitted text before replacing the canvas.
- Verify brush, line, rectangle, color, stroke-size, clear, save/load, undo/redo, mascot loading, and high-DPI canvas behavior remain unchanged after adding the Text tool.
- Resize the Tauri window while editing and after committing text; confirm the editor remains clamped to the board, committed text follows existing bitmap scaling, and no duplicate snapshot is produced. Check responsive layouts above 940 px, from 561–940 px, and at 560 px or narrower. There is no automated test, lint, Vitest, Playwright, or Rust test suite configured in the repository.

## Risks

- Rasterized text cannot be selected, moved, or restyled after commit; supporting that later would require a retained object model and a project format beyond PNG.
- DOM editor and Canvas 2D font metrics can vary across Windows WebView2 and other development platforms. The implementation must use the same computed font family, weight, style, size, and explicit line height to minimize preview-to-output shifts.
- The toolbar is already space-constrained and changes layout at 940 px and 560 px, so adding another tool and formatting controls may obscure the scene or overflow without explicit breakpoint testing.
- Tool switches, resize events, save, clear, load, undo, and redo can race with an active editor unless all completion and cancellation paths use centralized, idempotent session helpers.
- The existing Rust history implementation does not clear the redo stack when a new snapshot is pushed after undo. Text should use the existing history behavior without broadening this issue into an unrelated undo/redo refactor, but this pre-existing behavior may appear during manual testing.
- The repository has no automated frontend or integration test framework, so regressions in pointer interaction, keyboard input, responsive layout, and Tauri history depend on thorough manual validation.

## Open questions

   A. Rasterize on commit; edit by undoing and retyping, preserving the current PNG-only model. (recommended)
   B. Keep text as editable objects, requiring a document model, selection behavior, serialization format, and composed PNG export.
2. How should users enter and finish text on the board?
   A. Click-to-place inline multiline editor; Ctrl/Cmd+Enter commits, Escape cancels, and explicit line breaks are preserved. (recommended)
   B. Click-to-place single-line editor; Enter commits and Escape cancels.
   C. Type in a toolbar field, then click the board to place the prepared text.
