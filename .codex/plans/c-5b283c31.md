# Codex plan for mehrangta/funny-whiteboard-paint#6

Job: `c-5b283c31` · Revision: 1

Treat the issue as a request to make the Windows x64 release build an explicit package.json interface rather than leaving the Tauri command embedded in release.yml. Add a dedicated build:windows-x64 script, use it from both release and CI workflows to prevent command drift, document the same local command, and preserve the existing 0.1.0 version, Bun lockfile, stable-tag validation, unsigned executable format, and draft-only publication behavior.

## Steps

1. **Add the package-level Windows release build** — Add a `build:windows-x64` entry under `package.json` scripts with the exact command `tauri build --target x86_64-pc-windows-msvc --no-bundle`. Preserve the existing `dev`, `build`, `preview`, and `tauri` interfaces and make no dependency, package name, privacy, or version changes. Do not add a `packageManager` field because `.bun-version` already pins Bun 1.3.14 for the workflows. A scripts-only manifest change should not alter `bun.lock`; reject incidental dependency resolution changes.
   - Likely files: `package.json`
2. **Route automated builds through the package script** — Replace the inline `bun run tauri build --target $env:WINDOWS_TARGET --no-bundle` invocation in `release.yml` with `bun run build:windows-x64`. Apply the same replacement in `ci.yml` so pull-request validation and tag releases build through one package-owned interface. Retain `WINDOWS_TARGET` because it is still used by Rust target installation, `cargo check`, and executable path resolution. Do not change tag triggers, permissions, concurrency, cache behavior, version checks, executable staging, artifact naming, or GitHub release creation.
   - Likely files: `.github/workflows/release.yml`, `.github/workflows/ci.yml`
3. **Keep release data flow and failures unchanged** — The tag remains the release input: `vX.Y.Z` is converted to `X.Y.Z`, compared case-sensitively with `package.json`, `src-tauri/tauri.conf.json`, and the `funny_whiteboard` Cargo package, and rejected before build when any value differs. The new package script must propagate Tauri's nonzero exit status so frontend compilation, Rust compilation, target setup, or missing executable failures stop the job. Successful builds must still produce `src-tauri/target/x86_64-pc-windows-msvc/release/funny_whiteboard.exe`, which the existing PowerShell script stages as `funny-whiteboard-X.Y.Z-windows-x64.exe`. Release publication remains outside package.json and continues to create or update a draft containing exactly one asset.
   - Likely files: `package.json`, `.github/workflows/release.yml`, `.github/scripts/prepare-windows-executable.ps1`, `src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml`
4. **Document the shared local build interface** — Update the README Windows x64 build example to use `bun run build:windows-x64` and leave the documented output path and unsigned portable-executable limitations unchanged. Keep the generic `bun run tauri build` instructions for developers who intentionally want all Tauri bundle targets. Continue documenting that all three application versions must match before tagging and that the legacy `1.0` tag must not be reused.
   - Likely files: `README.md`
5. **Validate and roll out safely** — First validate JSON parsing, frozen-lockfile installation, and the frontend build. Then validate the Rust target and package script on Windows, checking the expected executable path. Let normal CI exercise the same script on a pull request. Merge without creating a release. For an actual release, update `package.json`, `src-tauri/Cargo.toml`, and `src-tauri/tauri.conf.json` together to an explicitly chosen stable SemVer, create a new matching `vX.Y.Z` tag, inspect the generated draft and executable on a clean Windows x64 machine, and publish manually only after acceptance. Do not use a tag merely to test the workflow because it has GitHub release side effects.
   - Likely files: `package.json`, `bun.lock`, `src-tauri/Cargo.toml`, `src-tauri/tauri.conf.json`, `.github/workflows/ci.yml`, `.github/workflows/release.yml`

## Validation

- Run `bun -e 'const p = await Bun.file("package.json").json(); if (p.scripts["build:windows-x64"] !== "tauri build --target x86_64-pc-windows-msvc --no-bundle") process.exit(1)'` to validate the manifest and exact script contract.
- Run `bun install --frozen-lockfile` and confirm `bun.lock` remains unchanged.
- Run `bun run build` to execute the repository's TypeScript and Vite frontend validation.
- On Windows with the configured Rust target, run `cargo check --manifest-path src-tauri/Cargo.toml --locked --target x86_64-pc-windows-msvc`.
- On Windows, run `bun run build:windows-x64` and verify `src-tauri/target/x86_64-pc-windows-msvc/release/funny_whiteboard.exe` exists and is non-empty.
- Open a pull request and confirm the existing `CI / Windows x64 executable` job completes, including executable staging and artifact upload. The repository has no unit-test or workflow-lint configuration to run separately.

## Risks

- The issue does not specify a release version. Changing `0.1.0` without an explicit target would invent release intent, so this plan deliberately adds the package interface without bumping any version.
- The Windows target is represented both by the package script and the workflows' `WINDOWS_TARGET` value. A future architecture change must update the package script, workflow target setup and output paths, and README together.
- The current planning environment has Bun 1.3.14 but no Cargo or Rust compiler, and it is not Windows; final Rust and executable validation must therefore run in the repository's Windows GitHub Actions job or an equivalent Windows development environment.
- The tag-triggered workflow writes GitHub release state. Testing with a real tag can create or overwrite a draft asset, while an existing published release intentionally causes failure.
- A scripts-only `package.json` edit should not regenerate dependencies. Unexpected `bun.lock` changes could introduce unrelated upgrades and should be excluded from this issue.
- Naming the script `build:windows-x64` is intentional: it builds the artifact but does not create a GitHub release. A generic `release` script would misleadingly imply that local execution performs the workflow's validation and GitHub draft-release steps.
